PROBLEMAS CRÍTICOS ENCONTRADOS
================================================================================

--- 1. SEGURIDAD — Gravedad: CRÍTICA ---

1.1 Login hardcodeado:
    - El login valida user === 'admin' && pass === 'admin123' directamente en el código
    - No consulta la base de datos
    - No usa hash de contraseña
    - Cualquier persona que lea el código fuente obtiene acceso total

1.2 Sin autenticación real:
    - Solo usa localStorage.setItem('sgp_auth', 'true') para mantener sesión
    - Cualquier usuario puede abrir DevTools y setear este valor
    - No hay verificación server-side de sesión
    - No hay expiración de sesión

1.3 Sin middleware de autenticación:
    - Ningún endpoint API verifica si el request viene de un usuario autenticado
    - Cualquier persona puede hacer curl/Postman a /api/personnel, /api/incidents, etc.
    - No hay validación de permisos por rol

1.4 Tabla users duplicada:
    - El CREATE TABLE users se ejecuta DOS veces con esquemas diferentes
    - La primera: (id, username, password, role, last_login)
    - La segunda: (id, username, name, role, permissions, last_login) — SIN password
    - La segunda sobreescribe la primera, perdiendo la columna password

1.5 Sin validación de input en el servidor:
    - No hay sanitización de datos entrantes
    - Vulnerable a XSS (Cross-Site Scripting)
    - Vulnerable a inyección SQL (aunque better-sqlite3 usa prepared statements, no hay validación de tipos)
    - No hay límite de tamaño en los campos de texto

1.6 Sin protección CSRF:
    - No hay tokens CSRF en los formularios
    - Un sitio malicioso podría hacer requests en nombre del usuario

1.7 Dependencias de seguridad instaladas pero NO usadas:
    - jsonwebtoken está en package.json pero nunca se importa
    - bcryptjs está en package.json pero nunca se importa
    - Se incluyeron explícitamente para auth pero nunca se implementaron

1.8 Tokens sensibles versionados:
    - Archivos TOKEN_DE_SINCRONIZACION.txt, TOKEN_FINAL_VERIFICACION.txt, VERSION_PARA_CEMENTAR.txt en el repo
    - Contienen tokens/credenciales que no deberían estar en control de versiones

--- 2. BASE DE DATOS — Gravedad: ALTA ---

2.1 Tabla users duplicada:
    - Ver punto 1.4 — causa pérdida de datos y esquema inconsistente

2.2 Migraciones con try/catch silencioso:
    - Los ALTER TABLE están envueltos en try/catch vacíos
    - Si una migración falla, no hay log ni notificación
    - Imposible saber qué columnas existen realmente en producción
    - Ejemplo: try { db.exec("ALTER TABLE personnel ADD COLUMN birthdate DATE;"); } catch(e) {}

2.3 Sin índices:
    - No hay índices en columnas frecuentemente consultadas
    - personnel_id en attendance, guard_shifts, personnel_records
    - date en guard_shifts, attendance, fuel_log
    - status en incidents, fleet, personnel
    - Esto causará degradación de performance con datos crecientes

2.4 Sin endpoints DELETE/PUT para la mayoría de entidades:
    - No se puede eliminar ni editar personal
    - No se puede eliminar ni editar incidencias
    - No se puede eliminar ni editar pagos
    - No se puede editar reservas
    - Solo existe DELETE para guard-shifts

2.5 Sin ON DELETE CASCADE en las Foreign Keys:
    - Las FK no tienen comportamiento de eliminación definido
    - Si se elimina un registro de personnel, los registros en attendance, guard_shifts, personnel_records quedan huérfanos
    - Datos inconsistentes garantizados con el tiempo

2.6 Endpoint de alertas de vencimientos vacío:
    - GET /api/alerts/vencimientos devuelve siempre { expirations: [], critical: 0 }
    - No hay lógica de verificación real
    - No consulta la tabla documents ni ninguna otra

2.7 Balance de finanzas hardcodeado:
    - GET /api/finances/balance devuelve valores fijos:
      { balance: 150000.50, monthly_income: 45000, monthly_expense: 12000 }
    - No calcula nada real desde la tabla payments

2.8 Endpoint de guardia vacío:
    - GET /api/guardia devuelve siempre { data: [] }
    - No consulta duty_log ni attendance

--- B. GUARDIAS — Propuestas Prioritarias ---

B1. VINCULACIÓN GUARDIA-ASISTENCIA-LIBRETA
    Descripción: Unificar los tres módulos en un flujo coherente
    Modelo relacional propuesto:
      guard_shifts → tiene un duty_log (libreta de guardia)
      guard_shifts → tiene múltiples attendance records
      duty_log → referencia al guard_shift actual
      attendance → referencia al guard_shift correspondiente
    Modificaciones a la DB:
      ALTER TABLE duty_log ADD COLUMN guard_shift_id INTEGER REFERENCES guard_shifts(id);
      ALTER TABLE attendance ADD COLUMN guard_shift_id INTEGER REFERENCES guard_shifts(id);
    Flujo integrado:
      1. Al iniciar turno → crear guard_shift + duty_log + check-in masivo del personal asignado
      2. Durante el turno → agregar novedades a la libreta vinculada al turno activo
      3. Al cerrar turno → check-out masivo + cierre de libreta + generación de resumen + traspaso
    Nuevos endpoints:
      - GET /api/guard-shifts/active → turno actual con toda la info asociada
      - POST /api/guard-shifts/:id/start → iniciar turno (check-in masivo)
      - POST /api/guard-shifts/:id/close → cerrar turno (check-out masivo + resumen)
      - POST /api/guard-shifts/:id/novedad → agregar novedad durante el turno

B2. VALIDACIONES DE GUARDIA
    Descripción: Implementar validaciones en backend y frontend
    Validaciones backend:
      - No asignar mismo bombero a dos turnos el mismo día
        SQL: SELECT COUNT(*) FROM guard_shifts WHERE personnel_id = ? AND date = ?
      - Mínimo de personal por turno (configurable en settings, default: 3)
      - Al menos 1 CHOFER certificado por turno
        SQL: JOIN con personnel_specialties WHERE specialty = 'CHOFER'
      - Al menos 1 OFICIAL o CABO por turno
        SQL: JOIN con personnel WHERE rank IN ('OFICIAL', 'CABO', 'CABO 1RO', 'SARGENTO')
      - No asignar personal en estado LICENCIA o BAJA
        SQL: JOIN con personnel WHERE status = 'ACTIVO'
      - No asignar personal con certificado médico vencido
        SQL: JOIN con personnel WHERE medical_expiry > date('now')
      - No asignar personal con licencia activa
        SQL: NOT EXISTS en licenses WHERE status = 'APROBADA' AND date overlap
    Validaciones frontend:
      - Mostrar advertencias antes de confirmar asignación
      - Resaltar conflictos en el calendario
      - Indicadores de cobertura: verde (completa), amarillo (mínima), rojo (insuficiente)

      CREATE TABLE rotation_config (
        id INTEGER PRIMARY KEY AUTOINCREMENT,
        min_per_shift INTEGER DEFAULT 3,
        require_chofer BOOLEAN DEFAULT 1,
        require_oficial BOOLEAN DEFAULT 1,
        shift_types TEXT DEFAULT 'MAÑANA,TARDE,NOCHE',
        exclude_weekends BOOLEAN DEFAULT 0,
        balance_strategy TEXT DEFAULT 'EQUITATIVO' -- EQUITATIVO, POR_ANTIGÜEDAD, POR_ESPECIALIDAD

    Algoritmo de distribución:
      1. Obtener lista de personal ACTIVO con especialidades

         d. Balancear guardias nocturnas equitativamente
      5. Generar cronograma completo para el mes
      6. Permitir ajustes manuales después de la generación
      7. Notificar a cada bombero su asignación

      POST /api/guard-shifts/auto-generate

    Controles adicionales:

RESUMEN EJECUTIVO
================================================================================

El sistema SGP-B Chimpay v2.0.4 tiene una base funcional sólida con un diseño
visual atractivo y una buena cobertura de módulos para la gestión de un cuartel
de bomberos. Sin embargo, presenta problemas críticos que impiden su uso en
producción:

1. SEGURIDAD: La autenticación es hardcodeada (admin/admin123), las APIs no
   tienen protección, y las dependencias de seguridad (JWT, bcrypt) están
   instaladas pero sin implementar.

2. DATOS DESCONECTADOS: La libreta de guardia, la asistencia y el calendario
   son módulos separados sin relación entre sí. No se puede saber quién estaba
   de servicio durante un incidente.

3. FUNCIONALIDADES INCOMPLETAS: Las alertas están vacías, el balance de
   finanzas es hardcodeado, el CRUD de personal es solo Create, y el mapa
   es un mock visual.

Las 3 mejoras de mayor impacto serían:

  a) AUTENTICACIÓN REAL + middleware de auth
     → Sin esto, el sistema no es seguro ni usable en producción

  b) UNIFICAR EL MÓDULO DE GUARDIAS
     → Conectar libreta, asistencia y calendario en un flujo coherente
     → Agregar validaciones y rotación automática

  c) IMPLEMENTAR EL SISTEMA DE ALERTAS
     → Es la funcionalidad más visible que está completamente vacía
     → Motor de alertas automáticas + notificaciones multi-canal

Con estas 3 mejoras, el sistema pasaría de ser un prototipo funcional a una
herramienta operativa real para un cuartel de bomberos.
